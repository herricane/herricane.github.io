Algorithms: Union Find

Hetopia
2020-06-07
Algorithms, Java

Recently I am taking a course of algorithms from Princeton here on Coursera. From now on I will take notes of this course and put them in the blog for the sake of revision.

Dynamic Connectivity

Dynamic connectivity problem is that given a set of N objects and a union command that connects two objects, make a find/connected query that determines whether there is a path connecting certain two objects. An example is shown in the following figure.

Here we define “is connected to” is an equivalence relation:

  • Reflexive: p is connected to p.
  • Symmetric: if p is connected to q, then q is connected to p.
  • Transitive: if p is connected to q and q is connected to r, then p is connected to r.

Connected components are the maximal sets of objects that are mutually connected, as shown in the figure.

This problem can be specified as a Java class called UF (union find) which contains two methods, one to implement union and the other to check connectivity. Note that the number of objects and operations can be huge, so an efficient algorithm is desired.

Quick-find

Quick-find is an algorithm for solving dynamic connectivity problem, and it is a so-called “eager algorithm”.

  • Data structure: an integer array indexed by objects.
  • Interpretation: two objects are connected if and only if their entries in the array are the same.

In this example 0, 5 and 6 has the same id, so these three objects are connected. It is similar for the rest.

For the find operation, we just need to check if p and q have the same id. For the union of p and q, we need to merge components containing p and q by changing all entries whose id equals id[p] to id[q].

Implementation

Complexity

algorithm initialize union find
quick-find N N 1

The quick-find algorithm takes N accesses to initialize the array, N for union command and 1 for find. That is to say, it takes N2 array accesses to process sequence of N union commands on N objects, which is too expensive. Generally speaking, quadratic time is much too slow for large problems.

Quick-union

Quick-union is another algorithm to solve dynamic connectivity problem and it’s called “lazy approach” to algorithm design where we avoid doing thing until we have to.

  • Data structure: an integer array id[] of size N.

  • Interpretation: id[i] is parent of i.

For the find operation we should check if p and q have the same root. For the union of p and q we should set the id of p‘s root to the id of q‘s root to merge two components.

Implementation

Complexity

algorithm initialize union find
quick-find N N 1
quick-union N N (including cost of finding roots) N (worst case)

The quick-union algorithm is faster but still too slow for some cases. The defects are that trees can grow too tall and for the worst case find can include N array accesses.

Quick-union Improvements

Improvement 1: Weighting

Weighted quick-union can avoid tall trees by keeping track of the size of each tree. We can balance the trees by always linking root of smaller trees to root of larger trees.

  • Data structure: Same as quick-union, but maintain extra array sz[i] to count number of objects in the tree rooted at i.

The find operation is the same as quick-union. The union operation is modified to:

  • Link root of smaller tree to the root of larger tree.
  • Update sz[i] array.

Improvement 2: Path Compression

Quick-union with path compression is that after computing the root of p, set the id of each examined node to point to that root.

Implementation

We can make every other node in the path point to its grandparent thereby halving path length.

Complexity

In theory WQUPC is not quite linear, but in practice it is linear.

QU + path compression N + M log N

Assignment 1: Percolation

The specification of the assignment is here.

My submission is here.

The basic idea is to treat every site as an element in the dynamic connectivity problem and use weighted QU algorithm. We need to create a virtual top site and a virtual bottom site so that we can check if the system percolates by checking whether the virtual top is connected to the virtual bottom.

However there is a backwash issue when we address the problem this way. Backwash means that some sites may be mistakenly considered as a full site. (A full site is an open site that can be connected to the top row via a chain of neighboring open sites.) This is because when the system percolates, the virtual top and the virtual bottom are connected, and in this case if a site is connected only to the bottom (but not the top), it will still be considered as connected to the top, making wrong judgment about whether it is a full site. This figure illustrates the backwash issue.

In order to solve this problem, we can have two WeightedQuickUnionUF objects instead of one: One has n+2 elements including the virtual top and bottom, and the other one has n+1 elements that includes only the virtual top. When checking if a system percolates, we use the former one; and when checking if a site is a full site, we use the latter one. In this way we solve the problem with more space occupied.
